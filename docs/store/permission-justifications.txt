================================================================
SINGLE PURPOSE
================================================================

Decant has one purpose: convert a document to Markdown at the moment the user
uploads it to an LLM chat site, so the chat receives lighter Markdown instead of
the original file.

Everything in the extension serves that single job — detecting an upload (file
picker, drag-and-drop, or paste) on a site the user has enabled, converting the
file inside the browser, and substituting the result back into that same upload.
The options page exists only to configure this: which sites it runs on, what
happens to each file type, and the hotkey that forces one upload through
unconverted.

Decant has no second feature. It does not browse, search, or collect, and it
does nothing at all on sites the user has not explicitly enabled. It transmits
user provided files only if the user specifically configures the settings to do
so.


================================================================
SCRIPTING JUSTIFICATION
================================================================

The scripting permission is used only to register content scripts dynamically:
registerContentScripts, updateContentScripts, unregisterContentScripts, and
getRegisteredContentScripts. Decant never calls executeScript, and never injects
remote or generated code.

This is what implements the extension's default-deny model. Rather than
declaring static content scripts for every supported site, Decant registers its
interception script only for hosts the user has explicitly enabled and for which
Chrome has granted host permission. Disabling a host unregisters it; with no
hosts enabled, no content script is registered anywhere.

Two scripts are registered per enabled host: the interceptor, and a small
MAIN-world shim that hooks file-input creation for sites whose upload button
uses a detached input element (for example kimi.com) that isolated-world
listeners cannot observe. Both are bundled in the package.


================================================================
STORAGE JUSTIFICATION
================================================================

Storage holds the user's own settings, and nothing else.

chrome.storage.sync holds the configuration edited on the options page: which
sites Decant is enabled on, the per-file-type routing rules, the passthrough
hotkey binding, and the saved default answer for the convert-or-not prompt. It
uses sync storage so these settings follow the user across the Chrome browsers
they are signed in to, the same way bookmarks do.

chrome.storage.local holds one number: a running total of estimated tokens
saved, displayed on the options page. It is kept out of sync storage because it
is written after every conversion, and sync storage has an 8KB-per-item quota.

No document content, file names, page content, or browsing activity is stored.
Nothing in storage is transmitted anywhere — Decant operates no server.


================================================================
HOST PERMISSION JUSTIFICATION
================================================================

Decant intercepts a file upload inside the page and substitutes a converted
version, so it has to run on the chat site where the upload happens. The four
bundled hosts — claude.ai, chatgpt.com, gemini.google.com and www.perplexity.ai
— are the LLM chats enabled by default.

The optional_host_permissions entries are never requested as wildcards. They
exist so a user can add their own chat host on the options page; the extension
then calls permissions.request() with that one concrete origin only, as
https://<host>/*. Nothing broader is ever granted, nothing beyond the four
bundled hosts is granted at install, and removing a host revokes its permission.

The declared surface is deliberately narrow: https://*/* rather than *://*/*,
so plain HTTP is out of scope, plus http://localhost/* and http://127.0.0.1/*,
needed only for the optional companion service — it runs on the user's own
machine and is reachable no other way.


================================================================
REMOTE CODE  —  answer: No, I am not using remote code
================================================================

Not a 1000-character field; recorded here as the verified basis for the
dashboard answer (checked 2026-07-21 against the packaged build).

- Every executable asset resolves through browser.runtime.getURL() to the
  packaged extension root (src/convert/assets.js). The pdf.js worker, the WASM
  decoders, the standard fonts and the ICC profiles all ship in the package;
  none is fetched from the network.
- Decant's own source contains no eval(), no new Function(), and no dynamic
  import().
- isEvalSupported: false is passed to pdf.js explicitly, and no custom
  content_security_policy is declared, so the MV3 default CSP applies.
- The only external URL in the extension is the GitHub bug-report link, opened
  as a tab navigation, never loaded as code.
- The companion/http transport receives data (text or JSON) that becomes
  document content. It is never evaluated, and it is unreachable unless the
  user configures an endpoint.

Two things a reviewer grepping the package may find, both benign:

- One eval( survives in the bundled third-party dependency code, inside a
  toFastProperties() helper, placed after an unconditional return. It is
  statically unreachable — the well-known V8 fast-properties idiom.
- pdf.js's wasm/ directory ships a quickjs-eval JavaScript interpreter for
  scripts embedded in PDF forms. Decant excludes it at build time (build.mjs):
  its only consumer is pdf.sandbox.mjs, which is never bundled, so it could
  never load. It is not in the uploaded package.

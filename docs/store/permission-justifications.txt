================================================================
SINGLE PURPOSE  (revised for 0.3.1 — page capture added)
================================================================

Decant has one purpose: turn content into Markdown at the moment the user sends
it to an LLM chat site, so the chat receives lighter Markdown instead of a
heavier original.

That job runs in two directions. Uploads: on sites the user has enabled, an
attached document (file picker, drag-and-drop, or paste) is converted inside
the browser and substituted into that same upload. Page capture: when the user
invokes Decant on a page they are reading (toolbar click, keyboard shortcut, or
context-menu pick), that page is converted to Markdown and attached in the chat
they last used.

Everything else serves that job — the options page only configures which sites,
what happens to each file type, and the hotkeys. Decant has no second feature.
It does not browse, search, or collect; it does nothing on sites the user has
not enabled, and reads no page without an explicit user gesture. It transmits
user files only if the user configures an endpoint.


================================================================
SCRIPTING JUSTIFICATION  (revised for 0.3.1 — executeScript now used)
================================================================

The scripting permission does two jobs, both user-controlled, executing only
code bundled in the package — never remote or generated code.

1) Dynamic content-script registration (register/update/
unregisterContentScripts): the default-deny model. The upload interceptor and
a small MAIN-world shim (for detached-picker sites like kimi.com) are
registered only for hosts the user enabled and Chrome granted. Disabling a
host unregisters it; with no hosts enabled, nothing is registered anywhere.

2) executeScript, for page capture only: when the user invokes capture with a
gesture, the packaged serializer bundle is injected into that page under the
activeTab grant to read its rendered content, and a small notice reports
progress and outcomes there. On enabled chat hosts it also places the captured
Markdown on the clipboard when the chat accepts no file attachments. Injected
files ship in the package; injected functions come from the packaged
background script.


================================================================
STORAGE JUSTIFICATION
================================================================

Storage holds the user's own settings, and nothing else.

chrome.storage.sync holds the configuration edited on the options page: which
sites Decant is enabled on, the per-file-type routing rules, the passthrough
hotkey binding, and the saved default answer for the convert-or-not prompt. It
uses sync storage so these settings follow the user across the Chrome browsers
they are signed in to, the same way bookmarks do.

chrome.storage.local holds two small machine-local values: a running total of
estimated tokens saved (displayed on the options page), and the hostname of the
enabled chat site that last received files — used only as the default
destination for page capture when no chat tab is open. Both are kept out of
sync storage because they are frequently written device state.

No document content, file names, page content, or browsing activity is stored.
Nothing in storage is transmitted anywhere — Decant operates no server.


================================================================
HOST PERMISSION JUSTIFICATION
================================================================

Decant substitutes converted uploads inside the chat page, so it must run
where uploads happen. The four bundled hosts — claude.ai, chatgpt.com,
gemini.google.com, www.perplexity.ai — are the defaults.

The optional_host_permissions entries are never requested as wildcards: adding
a chat host on the options page requests that one concrete origin only
(https://<host>/*). Nothing beyond the four bundled hosts is granted at
install, and disabling or removing a host revokes its permission.

The declared surface is deliberately narrow: https://*/* rather than *://*/*
(plain HTTP out of scope), plus http://localhost/* and http://127.0.0.1/*
solely for the optional companion service on the user's own machine.

The same grants also serve page-capture delivery: URL-pattern tabs.query finds
the user's most recent enabled-chat tab (Decant holds no "tabs" permission),
and the converted page is injected there. Captured pages need no host
permission — they are read once under activeTab.


================================================================
ACTIVETAB JUSTIFICATION  (new in 0.3.1)
================================================================

activeTab is the entire permission design for page capture. Capture runs only
from an explicit user gesture — a toolbar click, the capture keyboard shortcut,
or a context-menu pick — and the activeTab grant that gesture produces is what
lets Decant read the one page the user invoked it on, once. The packaged
serializer is injected into that tab to read its rendered content; the result
is converted to Markdown locally and attached in the user's chosen chat. The
same grant shows a small status notice on that page (capturing / delivered /
failure reason), so the user is never left guessing.

Decant requests no host permission for captured pages: no wildcard, no
per-origin prompts, no access before or after the gesture, and no reading of
any tab the user did not invoke it on. Nothing read from a captured page is
transmitted anywhere; conversion happens inside the browser.


================================================================
CONTEXTMENUS JUSTIFICATION  (new in 0.3.1)
================================================================

contextMenus renders the page-capture controls as native browser UI instead of
UI injected into pages. One parent item ("Decant page to…") lists the chat
sites the user has enabled — picking one captures the current page to that
chat instead of the default last-used one — plus a single checkbox ("Include
page images") that mirrors the options-page toggle for attaching the page's
images. The menu is built only from the user's own enabled-sites list; a site
the user disabled never appears, and with no sites enabled there is no menu at
all. A menu click is also the user gesture that grants activeTab for that
capture. Nothing is tracked and no other items are shown.


================================================================
COMMANDS JUSTIFICATION  (new in 0.3.1)
================================================================

commands registers one keyboard shortcut, capture-page (suggested Alt+Shift+C,
rebindable by the user at chrome://extensions/shortcuts). It does exactly what
the toolbar click does — capture the page the user is on and attach it in
their last-used chat — under the activeTab grant the shortcut gesture
provides. No other shortcuts are registered.


================================================================
REMOTE CODE  —  answer: No, I am not using remote code
================================================================

Not a 1000-character field; recorded here as the verified basis for the
dashboard answer (checked 2026-07-21 against the packaged build).

- Every executable asset resolves through browser.runtime.getURL() to the
  packaged extension root (src/convert/assets.js). The pdf.js worker, the WASM
  decoders, the standard fonts and the ICC profiles all ship in the package;
  none is fetched from the network.
- Decant's own source contains no eval(), no new Function() and no dynamic
  import(). Bundled dependencies do contain such calls; see the audit below —
  all are packaged, none is reachable, and none evaluates anything fetched.
- isEvalSupported: false is passed to pdf.js explicitly, and no custom
  content_security_policy is declared, so the MV3 default CSP applies.
- The only external URL in the extension is the GitHub bug-report link, opened
  as a tab navigation, never loaded as code.
- The page-capture serializer (capture/inject.js, new in 0.3.1) injected via
  executeScript files: is a packaged asset exactly like the content scripts;
  the accompanying func: injections serialize functions from the packaged
  background script. Neither path can carry remote or user-supplied code.
- The companion/http transport receives data (text or JSON) that becomes
  document content. It is never evaluated, and it is unreachable unless the
  user configures an endpoint.

What a reviewer grepping content/intercept.js WILL find, and why none of it is
remote code (audited against the packaged zip, 2026-07-21):

- One eval( and eight new Function( occurrences, all inside bundled
  third-party libraries, all shipped in the package rather than fetched:
    * bluebird 3.4.7 (via mammoth, the DOCX reader) — its toFastProperties()
      helper places eval() after an unconditional return, the well-known V8
      fast-properties idiom, so it is statically unreachable; its method-caller
      and promise-inspection caches compile accessors the same way.
    * underscore 1.13.8 (via mammoth / lop / duck) — the _.template compiler.
    * setimmediate 1.0.5 (via jszip) — coerces a string handler.
  None of these can execute: the MV3 default CSP (script-src 'self') blocks
  eval and the Function constructor outright, and no custom
  content_security_policy is declared to relax it. They are optimization fast
  paths and template helpers that degrade to their interpreted fallbacks —
  which is why DOCX, XLSX and PPTX conversion all work in the shipped
  extension. Critically, no string these touch comes from outside the package:
  nothing is fetched, downloaded or received over the network and then run.
- pdf.js's wasm/ directory ships a quickjs-eval JavaScript interpreter for
  scripts embedded in PDF forms. Decant excludes it at build time (build.mjs):
  its only consumer is pdf.sandbox.mjs, which is never bundled, so it could
  never load. It is not in the uploaded package.
